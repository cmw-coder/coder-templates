# 代码生成规范

## 1. 任务处理流程
1. **需求分析阶段**
   - 仔细阅读用户输入的测试点需求
   - 检查是否存在关联的.topox和conftest.py文件
   
2. **资料调研阶段**
   - 必须查阅以下目录中的相关内容：
     * press/ - 业务参考资料
     * pypilot/ - 框架使用文档
     * KE/ - 基础动作库
     * test_example/ - 历史测试案例
         * pypilot/ - pypilot自研自动化框架API使用说明

3. **脚本生成阶段**
   - 基于调研结果编写测试脚本
   - 严格遵循pypilot框架规范
   - 保持与现有代码风格一致

4. **质量检查阶段**
   - review下脚本内容，特别是命令行等checkcommand的字段是否借鉴已有资料，是否准确
   - 使用pylint进行静态检查
   - 修复所有致命错误(Fatal级别)
   - 确保在虚拟环境中执行检查

5. **交付阶段**
   - 将脚本保存到test_scripts/三级目录
   - 输出最终脚本路径

## 2. 编码规范细则

### 2.1 文件管理
- 禁止生成空文件(如无用的__init__.py)
- 测试脚本命名格式：test_[模块]_[功能].py
- 保持文件名与业务模块/拓扑文件一致

### 2.2 脚本结构
```python
class Test[FeatureName]:
    @classmethod
    def setup_class(cls):
        """如果conftest.py存在且基础配置满足需求，则为空即可"""
                
        @classmethod 
    def teardown_class(cls):
        """必须在此处编写配置清除代码"""
                
    def test_[specific_case](self):
        """单个测试用例"""
        
```

### 2.3 设备引用规范
- 设备实例：`gl.DUTx` (来自.topox)
- 接口实例：`gl.DUTx.PORTx`

### 2.4 代码风格
- 必须使用pypilot提供的方法
- 变量命名清晰表达业务含义
- 风格严格遵从example例子

## 3. 约束条件

1. **需求边界**
   - 严格按用户指定测试点实现
   - 仅在KE中有相关案例时可适当扩展
   
2. **目录规范**
   - 生成的脚本必须位于test_scripts/三级目录
   - 保持目录结构与业务模块对应

3. **质量红线**
   - Pylint检查必须通过
   - 不允许存在Fatal级别错误
   - 必须包含完整的清理逻辑

## 4. 最佳实践

1. **复用策略**
   - 优先复用KE中的基础动作
   - 参考test_example中的类似案例
   - 保持与现有代码风格一致

2. **异常处理**
   - 清理操作必须可靠执行

## 5. 工具使用

```bash
# Pylint检查命令
source .venv/bin/activate
pylint [生成的脚本].py

# 执行顺序
1. review代码并修复
2. 静态检查(pylint)
```
